# AiMall 后端技术架构设计文档（Django + MySQL）

## 1. 设计目标与范围

### 1.1 目标
使用 `Python + Django + MySQL` 构建 AiMall 电商系统后端，支撑：
- 管理端（商品/分类/客户/推荐/预览）
- 商城端（首页、分类、搜索、推荐、购物车、商品详情）
- 用户行为轨迹采集（作为推荐依据）

### 1.2 本期范围（V1）
- 管理员登录（开发账号密码）
- 商品、分类、客户 CRUD
- 行为轨迹记录与查询
- 推荐结果查询（先规则算法）
- 商城首页商品流/分类树/推荐/购物车接口
- 商品详情页数据接口（即使前端先做静态页，也建议接口预留）

## 2. 技术栈与组件选型

### 2.1 核心选型
- Python 3.11+（建议）
- Django 4.2+（LTS）
- Django REST Framework（DRF）
- MySQL 8.0+
- ORM：Django ORM
- 认证：DRF Token / JWT（二选一，建议 JWT）
- 文件存储：本地存储（开发）/ 对象存储（生产）

### 2.2 推荐配套组件（建议）
- `djangorestframework`
- `djangorestframework-simplejwt`（如采用 JWT）
- `django-filter`（列表筛选）
- `Pillow`（图片处理）
- `mysqlclient` 或 `PyMySQL`
- `drf-spectacular`（接口文档，可选）

## 3. 系统总体架构

### 3.1 分层设计
- 表现层（API）：DRF ViewSet / APIView，负责鉴权、参数校验、返回格式统一
- 应用层（Service）：业务编排、事务控制、推荐计算、行为写入
- 领域模型层（Model）：Django ORM 模型
- 数据层（Repository 可选）：复杂查询封装（Django 中可先用 Manager/QuerySet）
- 基础设施层：文件存储、缓存、日志、配置

### 3.2 模块划分（Django Apps）
建议按业务域拆分：
- `apps.authx`：管理员认证与登录
- `apps.products`：商品、SKU、图片
- `apps.categories`：商品分类（3 级）
- `apps.customers`：客户信息
- `apps.behaviors`：用户行为轨迹
- `apps.recommendations`：推荐计算与推荐结果查询
- `apps.carts`：购物车
- `apps.mall`：商城首页聚合接口（可选聚合层）
- `apps.common`：通用响应、异常、分页、上传、工具类

说明：
- `mall` 可只做聚合 API，不承载底层实体。
- 推荐逻辑与行为轨迹建议独立 app，方便后续升级算法。

## 4. 数据库设计（V1）

### 4.1 核心实体关系（文字版 ER）
- `customer` 与 `customer_behavior`：一对多
- `category` 自关联：父子树结构（最多 3 级）
- `product` 与 `category`：多对一（V1 一个商品一个主分类）
- `product` 与 `product_sku`：一对多
- `product` 与 `product_image`：一对多
- `customer` 与 `cart_item`：一对多
- `product_sku` 与 `cart_item`：一对多
- `customer` 与 `user_recommendation`：一对多（按推荐批次）

### 4.2 数据表设计（建议）

#### 4.2.1 管理员表 `admin_user`
- `id` bigint PK
- `username` varchar(64) UNIQUE
- `password_hash` varchar(255)
- `nickname` varchar(64)
- `status` tinyint（1启用/0禁用）
- `created_at` datetime
- `updated_at` datetime

开发初始化账号：
- `username=admin`
- 原始密码 `123456`（数据库存储哈希值）

#### 4.2.2 商品分类表 `product_category`
- `id` bigint PK
- `name` varchar(100)
- `parent_id` bigint NULL（0/NULL 表示一级）
- `level` tinyint（1/2/3，系统计算）
- `icon` varchar(255) NULL
- `sort` int default 0
- `enabled` tinyint default 1
- `created_at` datetime
- `updated_at` datetime

约束建议：
- `(parent_id, name)` 唯一（可选，避免同层重名）
- `level <= 3`

#### 4.2.3 商品表 `product`
- `id` bigint PK
- `name` varchar(200)
- `subtitle` varchar(255) NULL
- `category_id` bigint
- `main_image` varchar(500)
- `default_spec_name` varchar(100)
- `default_price` int（单位：分）
- `sales_count` int default 0
- `view_count` int default 0
- `status` varchar(20)（DRAFT/ON_SHELF/OFF_SHELF）
- `sort` int default 0
- `summary` text NULL
- `detail_content` longtext NULL（后续富文本/HTML）
- `created_at` datetime
- `updated_at` datetime

#### 4.2.4 商品图片表 `product_image`
- `id` bigint PK
- `product_id` bigint
- `image_url` varchar(500)
- `sort` int default 0
- `created_at` datetime

#### 4.2.5 商品 SKU 表 `product_sku`
- `id` bigint PK
- `product_id` bigint
- `sku_code` varchar(100) NULL
- `spec_values` json（如 `{\"颜色\":\"黑色\",\"容量\":\"128G\"}`）
- `spec_name_text` varchar(255)（展示快照）
- `price` int（分）
- `stock` int
- `image_url` varchar(500) NULL
- `is_default` tinyint default 0
- `status` tinyint default 1
- `created_at` datetime
- `updated_at` datetime

#### 4.2.6 客户表 `customer`
- `id` bigint PK
- `name` varchar(100)
- `phone` varchar(20)
- `age` int NULL
- `hobby` varchar(255) NULL
- `address` varchar(500) NULL
- `avatar` varchar(500) NULL（可选）
- `last_active_at` datetime NULL
- `created_at` datetime
- `updated_at` datetime

#### 4.2.7 用户行为轨迹表 `customer_behavior`
- `id` bigint PK
- `customer_id` bigint
- `behavior_type` varchar(32)
  - `SEARCH`
  - `CLICK_CATEGORY`
  - `VIEW_PRODUCT`
  - `ADD_TO_CART`
- `target_type` varchar(32)（KEYWORD/CATEGORY/PRODUCT）
- `target_id` bigint NULL
- `target_name` varchar(255) NULL（关键词或对象名称快照）
- `source_page` varchar(50) NULL（HOME/PRODUCT_LIST/PRODUCT_DETAIL 等）
- `extra_data` json NULL
- `created_at` datetime

索引建议：
- `idx_behavior_customer_time (customer_id, created_at desc)`
- `idx_behavior_type_time (behavior_type, created_at desc)`

#### 4.2.8 购物车表 `cart_item`
- `id` bigint PK
- `customer_id` bigint
- `product_id` bigint
- `sku_id` bigint
- `quantity` int
- `checked` tinyint default 1
- `created_at` datetime
- `updated_at` datetime

唯一约束建议：
- `(customer_id, sku_id)` 唯一（同 SKU 加购合并数量）

#### 4.2.9 推荐结果表（可选缓存）`user_recommendation`
- `id` bigint PK
- `customer_id` bigint
- `product_id` bigint
- `score` decimal(10,4)
- `reason` varchar(255)
- `batch_no` varchar(64)
- `generated_at` datetime

说明：
- V1 也可以不落库，实时计算返回；若需要管理端查看“推荐时间/分值/原因”，建议落库或缓存快照。

## 5. Django 模型与业务规则设计

### 5.1 分类层级规则
- 保存分类时，根据 `parent_id` 计算 `level`
- 若父分类 `level=3`，禁止创建子分类
- 编辑分类时禁止将某分类移动到其子孙节点下（避免环）
- 删除分类前检查：
  - 是否存在子分类
  - 是否有关联商品

### 5.2 商品与 SKU 规则
- 商品至少 1 个 SKU
- `default_price`、`default_spec_name` 可由默认 SKU 同步生成
- 删除商品时联动删除 SKU、图片（事务处理）
- 商品上/下架时校验 SKU 库存（可选）

### 5.3 行为轨迹规则
- 商城端行为写入应轻量级，失败不影响主流程（但需日志记录）
- 轨迹数据保留 `target_name` 快照，避免后续对象名称变化影响推荐解释
- 推荐计算优先读取近 30 天行为（可配置）

### 5.4 购物车规则
- 同一用户同一 `sku_id` 再次加入购物车时数量累加
- 商品/sku 下架时购物车项保留，但在结算前需校验（V1 先提示）

## 6. API 设计（V1）

### 6.1 接口风格约定
- 前缀：`/api/v1/`
- 响应统一：
```json
{ "code": 0, "message": "success", "data": {} }
```
- 分页参数：`page`, `page_size`
- 列表返回：`list`, `page`, `page_size`, `total`

### 6.2 认证相关（管理端）
- `POST /api/v1/admin/auth/login`
  - 入参：`username`, `password`
  - 出参：token / refresh_token（如 JWT）
- `POST /api/v1/admin/auth/logout`
- `GET /api/v1/admin/auth/profile`

### 6.3 分类管理（管理端）
- `GET /api/v1/admin/categories/tree`
- `POST /api/v1/admin/categories`
- `PUT /api/v1/admin/categories/{id}`
- `DELETE /api/v1/admin/categories/{id}`

### 6.4 商品管理（管理端）
- `GET /api/v1/admin/products`
  - 支持筛选：`keyword`, `category_id`, `status`
- `GET /api/v1/admin/products/{id}`
- `POST /api/v1/admin/products`
- `PUT /api/v1/admin/products/{id}`
- `DELETE /api/v1/admin/products/{id}`
- `POST /api/v1/admin/products/{id}/status`
- `POST /api/v1/admin/upload/image`

### 6.5 客户管理（管理端）
- `GET /api/v1/admin/customers`
  - 支持筛选：`keyword`（姓名/手机号）
- `GET /api/v1/admin/customers/{id}`
- `POST /api/v1/admin/customers`
- `PUT /api/v1/admin/customers/{id}`
- `DELETE /api/v1/admin/customers/{id}`
- `GET /api/v1/admin/customers/{id}/behaviors`

### 6.6 智能推荐（管理端）
- `GET /api/v1/admin/recommendations`
  - 支持 `customer_id` / `keyword`
  - 返回每个用户的推荐商品列表、原因、分值、时间
- `POST /api/v1/admin/recommendations/refresh`
  - 支持按用户刷新或全量刷新（V1 可先按用户）

### 6.7 商城端接口

#### 6.7.1 首页与分类
- `GET /api/v1/mall/home`
  - 返回：首页聚合数据（分类树、推荐4个、随机商品20个）
  - 或拆分为多个接口，建议先拆分，便于缓存
- `GET /api/v1/mall/categories/tree`
- `GET /api/v1/mall/products/random?size=20&cursor=...`

#### 6.7.2 搜索与商品列表
- `GET /api/v1/mall/products`
  - 参数：`keyword`, `category_id`, `page`, `page_size`

#### 6.7.3 商品详情
- `GET /api/v1/mall/products/{id}`
  - 返回商品基础信息、图集、SKU、详情内容（本期可先简化）

#### 6.7.4 推荐
- `GET /api/v1/mall/recommendations?customer_id={id}&size=4&exclude_ids=1,2`
- `POST /api/v1/mall/recommendations/refresh`（可选，前端“换一换”也可直接 GET 新结果）

#### 6.7.5 购物车
- `GET /api/v1/mall/cart/items?customer_id={id}`
- `POST /api/v1/mall/cart/items`
- `PUT /api/v1/mall/cart/items/{id}`（改数量/勾选）
- `DELETE /api/v1/mall/cart/items/{id}`（可选）
- `POST /api/v1/mall/cart/settlement-preview`（V1 可占位）

#### 6.7.6 行为轨迹上报（商城端）
- `POST /api/v1/mall/behaviors`
  - 入参：`customer_id`, `behavior_type`, `target_type`, `target_id`, `target_name`, `source_page`, `extra_data`

## 7. 推荐系统设计（V1 规则算法）

### 7.1 设计原则
- 先可用，再优化
- 可解释（管理端可展示推荐原因）
- 依赖行为轨迹数据，不引入复杂训练流程

### 7.2 评分策略（示例）
按最近 30 天行为给商品或分类打分：
- 搜索关键词命中商品名称：+3
- 点击类别：该类别商品 +2
- 点击商品详情：该商品 +4
- 加入购物车：该商品 +6
- 时间衰减：越近权重越高（例如 7 天内 *1.2）

生成逻辑：
1. 读取用户近 30 天行为
2. 汇总兴趣分类、兴趣商品、兴趣关键词
3. 从商品池中召回候选商品（上架状态）
4. 按规则打分并排序
5. 过滤已推荐/已排除商品（支持“换一换”）
6. 返回 TopN，并附 `reason`

### 7.3 “换一换”实现建议
- 前端传 `exclude_ids`
- 后端从候选集中排除后继续返回 4 个
- 若不足 4 个，用同类商品随机补齐

### 7.4 推荐结果缓存（可选）
- 使用 Redis 或数据库快照缓存用户推荐结果 5~10 分钟
- 管理端查看时可读取最新快照

## 8. 安全设计

### 8.1 认证与鉴权
- 管理端接口必须鉴权（JWT/Token）
- 商城端 V1 若未做真实用户登录，可使用临时 `customer_id` + 基础校验，但管理端接口不可共用该策略

### 8.2 密码安全
- 管理员密码使用 Django 哈希（PBKDF2）
- 禁止明文存储 `123456`

### 8.3 输入校验
- DRF Serializer 做字段校验
- 分类层级、商品价格、库存、手机号等均需服务端校验

### 8.4 上传安全
- 限制图片类型与大小
- 重命名文件、防止路径穿越
- 建议记录上传日志（可选）

## 9. 性能与可扩展性

### 9.1 数据库优化
- 分类树接口：可缓存（变更后失效）
- 商品列表：按 `status/category_id/created_at` 建索引
- 行为轨迹：按 `customer_id + created_at` 建索引
- 购物车：按 `customer_id` 建索引

### 9.2 缓存建议（V1 可选）
- 分类树缓存（高频读取）
- 首页随机商品池（短时缓存）
- 推荐结果缓存（按用户）

### 9.3 接口优化
- 首页接口建议拆分：分类/推荐/商品流独立请求，便于并发与缓存
- 商品列表返回摘要字段，详情接口返回完整字段
- 图片使用缩略图字段，避免首页拉大图

## 11. 开发实施建议（与前端协同）

### 11.1 开发顺序
1. 先建基础工程（Django + DRF + MySQL）
2. 完成认证、统一响应、异常处理、分页
3. 完成分类 CRUD 与分类树接口（商城/后台共用数据基础）
4. 完成商品 CRUD（含 SKU、图片上传）
5. 完成客户 CRUD + 行为轨迹查询
6. 完成商城首页/搜索/商品详情/购物车接口
7. 完成推荐规则服务与推荐接口

### 11.2 联调优先级
- 第一优先：分类树、商品列表、客户列表（撑起页面开发）
- 第二优先：商品详情、行为上报、购物车
- 第三优先：推荐与“换一换”

## 13. 与 GhostFit 的复用边界说明

用户明确要求复用 `GhostFit` 前端资源，但后端语言不同，因此本项目后端复用方式为“接口契约与字段风格对齐”，而非代码复用：
- 参考 `GhostFit` 管理端前端的列表/抽屉页面交互，设计稳定接口字段
- 保持分页、列表、详情、树形结构接口返回格式一致或相近
- 为前端复用 `Schema*` 系列组件提供稳定 schema/数据结构支持
