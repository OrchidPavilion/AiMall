# AiMall 前端技术架构设计文档（PC 管理端 + PC 商城端）

## 1. 设计目标

- 在 `Vue + TypeScript` 技术栈下快速搭建可持续迭代的双端 Web 系统（管理端/商城端）。
- 强制复用 `GhostFit` 的成熟后台前端资源，降低重复开发成本。
- 统一 API 封装、类型定义、状态管理、主题变量、通用 CRUD 页面范式。
- 为后续扩展（订单、支付、权限、推荐算法升级）预留结构。

## 2. 总体架构方案

### 2.1 推荐工程组织（单仓库前端双应用）
建议在 `AiMall` 下采用前端子目录方式：
- `frontend-admin/`：PC 管理端（Vue3 + TS）
- `frontend-mall/`：PC 商城端（Vue3 + TS）

原因：
- 管理端和商城端 UI/交互差异明显，拆分构建与部署更清晰
- 便于分别上线与独立配置接口域名
- 仍可通过共享包/共享目录复用类型与 API 定义

### 2.2 技术选型
- 框架：Vue 3（Composition API）
- 语言：TypeScript
- 构建工具：Vite
- 路由：Vue Router
- 状态管理：Pinia
- HTTP：Axios（统一封装）
- UI 组件库：Element Plus（管理端）；商城端可继续使用 Element Plus + 自定义样式
- 样式：SCSS（主题变量 + 模块化页面样式）
- 工具库：`dayjs`、`lodash-es`（按需）

### 3.3 复用原则
- 优先“复制并改名复用”，不要重新设计一套 CRUD/抽屉规范。
- 先保留 `GhostFit` 通用组件 API，业务页面按 AiMall 字段重配 schema。
- 仅在功能不满足时做向后兼容扩展（不要直接破坏现有组件契约）。
- 所有扩展优先抽象到通用组件层，避免页面内重复实现。

### 4.2 路由与菜单设计（支持 3 级菜单）
采用 `GhostFit` 的 `meta + children` 路由结构，菜单从路由元信息生成。

建议路由示例（简化）：
```ts
[
  { path: '/login', meta: { hidden: true } },
  {
    path: '/', component: Layout, children: [
      {
        path: 'mall', meta: { title: '商城', icon: 'shop' }, children: [
          {
            path: 'product-manage', meta: { title: '商品管理' }, children: [

              { path: 'categories', meta: { title: '类别列表' } },

          {
            path: 'customer-manage', meta: { title: '客户管理' }, children: [
```

### 4.3 页面开发范式（直接复用 GhostFit 模式）

#### 4.3.1 列表页标准结构
- `ListPageScaffold`：容器 + 搜索区 + 操作区 + 表格 + 分页
- `SchemaSearchForm`：搜索项 schema 驱动
- `SchemaTable`：列表列 schema 驱动 + slot 自定义渲染
- `SchemaDrawerForm`：新增/编辑抽屉
- `DetailDrawerLayout`：详情抽屉（客户详情）

#### 4.3.2 推荐映射到 AiMall 页面
- 商品列表：复用 `GhostFit` `src/views/product/List.vue` 的分页/批量操作/状态切换模式
- 类别列表：复用 `GhostFit` `src/views/store/Categories.vue` 的 3 级树表格 + 抽屉 + 父级级联选择器模式
- 客户列表：复用 `GhostFit` `src/views/customer/List.vue` 的详情抽屉 + 行为轨迹时间线 + 表单抽屉模式
- 行为轨迹展示：复用 `BehaviorTimeline.vue`，仅替换数据字段映射

### 4.4 管理端页面模块设计

#### 4.4.1 登录页
- 固定账号密码登录（开发环境）：`admin / 123456`
- 登录成功存储 token（后端返回）
- 跳转后台首页（可先直达商品列表）

#### 4.4.2 商品列表页
- 列字段：主图、名称、分类、默认规格、价格、销量、浏览量、状态、操作
- 搜索条件：名称、分类、状态
- 支持图片缩略图展示、状态标签、批量操作
- 编辑页可采用独立页面（参考 GhostFit `product/Create.vue`、`product/Edit.vue`）

#### 4.4.3 类别列表页
- 树形表格显示 1~3 级分类
- 抽屉新增/编辑分类
- `el-cascader` 选择父分类，自动限制最大层级

#### 4.4.4 智能推荐页
- 用户筛选表单（姓名/手机号）
- 推荐结果表格（用户、推荐商品、推荐原因、分值、时间）
- 支持刷新推荐按钮

#### 4.4.5 线上预览页
- 显示商城 URL
- 一键打开商城（`window.open`）
- 可配置环境地址

#### 4.4.6 客户列表页
- 列表 CRUD（姓名、手机号、年龄、爱好、地址）
- 点击客户姓名弹出详情抽屉
- 详情抽屉含“基本信息 + 行为轨迹”页签

### 4.5 状态管理（Pinia）
- `auth.store.ts`：登录态、用户信息、登出
- `app.store.ts`：主题、侧边栏折叠、全局配置
- `product.store.ts`：商品编辑草稿、分类缓存（可选）
- `customer.store.ts`：客户详情缓存（可选）
- `recommendation.store.ts`：推荐页筛选状态/刷新状态（可选）

原则：
- 页面级查询条件优先使用本地 `reactive/ref`
- 跨页面共享、登录态、长期缓存才入 store

### 4.6 API 封装设计
沿用 `GhostFit` 模式：
- `request.ts`：Axios 实例、请求/响应拦截器、错误处理
- `modules/*.ts`：按领域拆分 API
- `index.ts`：统一导出

管理端 API 模块建议：
- `authApi`
  - `login`
  - `logout`
  - `getProfile`
- `productApi`
  - `getList` `getDetail` `create` `update` `delete`
  - `uploadImage`
  - `toggleStatus`
- `categoryApi`
  - `getTree` `create` `update` `delete`
- `customerApi`
  - `getList` `getDetail` `create` `update` `delete`
  - `getBehaviors`
- `recommendationApi`
  - `getUserRecommendations`
  - `refreshUserRecommendations`

### 4.7 类型定义建议（TypeScript）
- `Product`
- `ProductSku`
- `ProductCategory`
- `Customer`
- `CustomerBehavior`
- `RecommendationItem`
- `PagedResponse<T>`
- `ApiResponse<T>`

建议将 DTO 与表单类型分离：
- `ProductDTO` / `ProductFormModel`
- `CustomerDTO` / `CustomerFormModel`

### 4.8 样式与主题（蓝白风格）
复用 `GhostFit` 的 SCSS 变量与 Element Plus 主题覆盖，再按 AiMall 调整：
- 主色：蓝色系（按钮、链接、强调态）
- 背景：浅灰/白色卡片
- 边框：浅蓝灰细线
- 表格 hover、抽屉头部、标签颜色统一

建议新增变量：
- `$brand-primary`
- `$brand-primary-light`
- `$page-bg`
- `$card-bg`
- `$border-color`

### 5.3 商城端关键页面设计

#### 5.3.1 首页（Home）
模块拆分：
- `MallHeader`：搜索框 + 购物车 icon + 用户头像 icon
- `CategoryNav`：左侧 3 级分类导航
- `RecommendPanel`：4 个推荐商品 + “换一换”
- `ProductGrid`：随机商品流（每批 20 条）

交互要点：
- 分类一级 hover 显示 `CategoryFlyout`（平滑动画）
- 点击分类/搜索时写入行为轨迹接口
- 推荐“换一换”需携带 `excludeIds` 或 `pageToken`，避免重复
- 商品流滚动加载写入节流逻辑，避免重复请求

#### 5.3.2 商品列表页（ProductList）
- 支持来自搜索词/分类条件
- 使用网格布局（一排 4 个）
- 支持分页（优先）或无限滚动（可选）
- 点击商品进入详情并记录行为轨迹

#### 5.3.3 购物车页（Cart）
- 列表项支持勾选、改数量、删除（删除可 V1 可选）
- 底部汇总栏自动计算已选总价
- 结算按钮（占位）

#### 5.3.4 商品详情页（ProductDetail）
本期先做页面结构：
- 左侧图集/主图
- 右侧名称/价格/销量/浏览量/默认规格
- 加购按钮占位
- 详情内容区域占位

## 6. 前后端接口协同规范（前端视角）

### 6.1 响应格式建议（统一）
```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

分页结构建议：
```json
{
  "list": [],
  "page": 1,
  "pageSize": 20,
  "total": 100
}
```

### 6.2 上传图片约定
- 管理端商品图片上传走独立接口
- 返回 URL + 文件名 + 尺寸（可选）
- 前端统一在 `productApi.uploadImage()` 封装

### 6.3 行为轨迹上报时机（商城端）
- 搜索提交成功后上报 `SEARCH`
- 点击分类上报 `CLICK_CATEGORY`
- 打开商品详情上报 `VIEW_PRODUCT`
- 加入购物车成功后上报 `ADD_TO_CART`

## 7. 开发落地建议（按优先级）

3. 实现登录页（固定账号密码）
4. 打通商品分类/客户列表最小 CRUD 页面（先假数据或后端 stub）

4. 线上预览页面

### 7.3 第三阶段（商城端）

## 8. 风险与注意事项
- `GhostFit` 现有通用组件与 AiMall 字段模型不完全一致，需要做 schema 映射，不要在页面中写死逻辑。
- 商城端与管理端建议分应用，不要把商城页面硬塞进后台布局工程。
- 推荐“换一换”如果后端没有 token 机制，前端需用 `excludeIds` 做去重。
- 行为轨迹若不上报成功，会影响推荐结果；前端需容错但要记录日志。
